# 脑洞集 - 测试系统

> 测试工程师专用窗口管理

---

## 文件说明

| 文件名 | 说明 |
|--------|------|
| `bug-list.md` | Bug 跟踪主文件，包含所有 Bug 的详细信息和当前状态 |
| `test-progress-report-YYYY-MM-DD.md` | 每日测试进度报告 |

---

## Bug 状态说明

| 状态 | 标记 | 说明 |
|------|------|------|
| 未开始 | ⚪ | Bug 已发现但未开始修复 |
| 已修复，待验证 | 🟡 | 修复已完成，需要测试工程师验证 |
| 已验证，已关闭 | 🟢 | 测试验证通过，此 Bug 已关闭 |
| 未修复，重新打开 | 🔴 | 验证发现修复无效或引入新问题，此 Bug 重新打开 |

---

## 如何标记 Bug 为"已修复，待验证"

开发人员修复完成后，请修改 `bug-list.md` 中对应 Bug 的状态字段：

1. 找到对应的 Bug 条目
2. 将 `当前状态` 改为 `🟡 已修复，待验证`
3. 更新 `状态更新时间` 为当前日期
4. 可选：在 `测试备注` 中添加修复说明

示例：

```markdown
## BUG-001：CardEditor "存为模板"功能调用错误的API

| 项目 | 内容 |
|------|------|
| **bug编号** | BUG-001 |
| **严重程度** | 🔴 严重 |
| **优先级** | 高 |
| **当前状态** | 🟡 已修复，待验证 |
| **发现时间** | 2026-04-20 |
| **状态更新时间** | 2026-04-20 |
| **测试备注** | 已修复，将 window.api.template.create 改为 api.template.create |
```

---

## 验证流程

1. 当检测到有 Bug 标记为"已修复，待验证"时，测试工程师将：
   - 拉取最新代码
   - 按照 Bug 的复现步骤进行测试验证
   
2. 如果验证通过：
   - 将 Bug 状态改为 `🟢 已验证，已关闭`
   - 更新状态更新时间
   - 在测试备注中记录验证结果
   
3. 如果验证失败或引入新问题：
   - 将 Bug 状态改为 `🔴 未修复，重新打开`
   - 更新状态更新时间
   - 在测试备注中详细说明问题

---

## 每日报告

每日晚上，测试工程师将生成一份新的测试进度报告，记录：
- 总体进度统计
- Bug 分布情况
- 今日工作内容
- 待完成工作
- 严重 Bug 概览
