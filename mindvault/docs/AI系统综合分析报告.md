## 二、七大核心问题解答

### 问题1：记忆系统架构

#### 你的问题
记忆功能是全局还是仅限每个消息，又或者二者都有，分层记忆？

#### 答案
**全局记忆**，目前没有分层设计。

#### 详细说明

**记忆存储方式**

| 实现 | 文件 | 存储位置 | 状态 |
|------|------|----------|------|
| SQLite版 | `ai-memory-sqlite.ts` | 数据库 `ai_memories` 表 | **当前使用** |

**记忆类型（5种）**

```
┌─────────────────────────────────────────────────────────┐
│                    全局记忆池 (SQLite)                     │
├─────────────────────────────────────────────────────────┤
│  fact       │ 用户提到的事实信息（姓名、地点、事件）          │
│  preference │ 用户偏好（喜欢的颜色、风格、习惯）             │
│  style      │ 用户的写作/沟通风格                          │
│  goal       │ 用户的目标或计划                             │
│  context    │ 重要的上下文信息                             │
└─────────────────────────────────────────────────────────┘
```

**记忆工作流程**

```
用户发送消息
    ↓
AI分析对话内容
    ↓
自动提取关键信息（调用 extractMemories）
    ↓
存入全局记忆池（去重、评分）
    ↓
下次对话时，根据关键词检索相关记忆
    ↓
注入到 System Prompt 中
```

**大白话解释**

想象一下，AI有一个**"长期笔记本"**：

1. **每次聊天后**，AI会自动翻看刚才说了什么
2. **发现重要信息**（比如你说"我喜欢蓝色"），就记到笔记本里
3. **下次聊天时**，AI会先翻翻笔记本，看看有没有相关信息
4. **把相关记忆**加到回复的参考材料里

**关键点**：
- ✅ 记忆是**全局共享**的，不区分对话窗口
- ✅ 记忆会**自动提取**，不需要手动添加
- ✅ 记忆有**重要性评分**（1-10分）
- ❌ **没有分层设计**（比如短期记忆/长期记忆）
- ❌ **没有对话级记忆**（每个聊天窗口独立记忆）

### 问题2：RAG功能与创意读取能力

#### 你的问题
跟AI助手聊天的话，他有RAG功能吗？能读取我的创意内容吗？

#### 答案
**有RAG功能**，AI可以通过工具读取你的创意内容。但RAG需要手动开启索引。

#### 重要澄清：RAG vs 工具读取

这是两个**不同的能力**：

| 能力 | 方式 | 状态 | 说明 |
|------|------|------|------|
| **工具读取创意** | AI调用 `search_creativity` 等工具 | ✅ **完全可用** | 关键词搜索，无需配置 |
| **RAG语义检索** | 向量化后语义相似度搜索 | ⚠️ **需手动索引** | 理解语义后找相似内容 |

#### RAG系统架构

```
┌─────────────────────────────────────────────────────────┐
│                    RAG 知识库系统                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  创意内容 ──→ 文本分块 ──→ 向量化 ──→ SQLite存储          │
│                                ↓                         │
│  用户提问 ──→ 向量化 ──→ 相似度计算 ──→ 返回相关内容       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

#### RAG核心代码（`rag-service.ts`）

| 功能 | 方法 | 说明 |
|------|------|------|
| 向量化 | `getEmbeddingFromOpenAI()` | 使用 OpenAI text-embedding-3-small 模型 |
| 文本分块 | `chunkText()` | 最大500字一块，重叠50字 |
| 相似度计算 | `cosineSimilarity()` | 余弦相似度算法 |
| 索引创意 | `indexAllCreativities()` | 批量向量化所有创意 |
| 检索 | `search()` | 向量检索 + 关键词回退 |

#### AI读取创意的方式

**方式一：通过工具调用（主要方式，完全可用）**

AI可以调用以下工具来读取你的创意：

| 工具名 | 功能 | 示例 |
|--------|------|------|
| `search_creativity` | 搜索创意 | "帮我找关于'小说'的创意" |
| `list_creativities` | 列出创意 | "看看我最近写了什么" |
| `get_creativity_detail` | 获取详情 | "打开ID为xxx的创意" |
| `global_search` | 全局搜索 | "搜索所有内容" |

**方式二：通过RAG检索（需要手动索引）**

1. 用户需要先点击"索引创意库"按钮
2. 系统会把所有创意向量化存储
3. 对话时AI可以检索相关创意片段

#### 大白话解释

**工具读取** = AI直接翻看你的档案柜
- 你说"找关于小说的创意"，AI用关键词搜索
- **随时可用**，不需要额外配置

**RAG检索** = AI用智能索引系统找内容
- 你说"找恐怖氛围的内容"，AI理解语义后找相似内容
- **需要先建立索引**（点一个按钮），而且创意更新后索引不会自动刷新

**当前状态**：
- ✅ 工具调用方式**完全可用**
- ✅ RAG功能**代码已实现**
- ⚠️ RAG需要**手动开启索引**
- ❌ RAG**不会自动更新**（创意变更后需重新索引）
- ⚠️ RAG目前只支持**OpenAI模型**（Anthropic不支持embedding）

#### 实际对话示例

```
用户：帮我看看最近写了什么创意？

AI调用工具：list_creativities(page=1, pageSize=10)

AI回复：我帮你查了一下，你最近创建了5条创意：
1. 《恐怖无限流小说大纲》- 2026-05-20
2. 《主角人物设定》- 2026-05-19
3. ...
```

### 问题3：Function Calling 与 MCP 协议能力

#### 你的问题
有function calling能力和mcp协议吗？例如我想查高铁票，ai小助手是否能帮我直接查询出数据并回答？

#### 答案
**两者都支持**。查高铁票需要接入第三方MCP服务。

**工具调用流程**

```
用户："帮我创建一条创意，标题是'新想法'"
```

#### MCP 协议能力

**什么是MCP？**

MCP（Model Context Protocol）是一个开放协议，让AI可以连接外部工具服务。

```
┌─────────────────────────────────────────────────────────┐
│                    MCP 架构示意                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   MindVault AI ←──→ MCP Bridge ←──→ MCP Server         │
│                          ↑                              │
│                          │                              │
│              ┌───────────┼───────────┐                  │
│              ↓           ↓           ↓                  │
│         高铁查询      天气服务      其他API...            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**MCP支持情况**

| 功能 | 状态 | 说明 |
|------|------|------|
| MCP客户端 | ✅ 已实现 | `mcp-client.ts` |
| MCP桥接层 | ✅ 已实现 | `mcp-bridge.ts` |
| stdio传输 | ✅ 支持 | 本地进程通信 |
| HTTP传输 | ✅ 支持 | 远程API调用 |
| 工具发现 | ✅ 支持 | 自动获取工具列表 |
| 工具调用 | ✅ 支持 | 统一执行接口 |

#### 查高铁票的可行性分析

**现状**：系统**没有内置**高铁票查询工具。

**解决方案**：通过MCP接入第三方服务。

**方案一：接入12306 MCP Server（推荐）**

```javascript
// 配置 MCP Server
{
  "id": "12306-mcp",
  "name": "高铁票查询",
  "command": "node",
  "args": ["12306-mcp-server.js"],
  "enabled": true
}
```

配置后，AI就能：
```
用户：帮我查一下明天北京到上海的高铁票

AI调用MCP工具：query_train(from="北京", to="上海", date="2026-05-25")

AI回复：明天北京到上海的高铁有：
G1次 08:00-12:30 二等座¥553
G3次 09:00-13:30 二等座¥553
...
```

**方案二：通过联网搜索**

AI可以调用 `web_search` 工具，但这种方式**不如MCP精准**。

#### 大白话解释

**Function Calling** 就像给AI配了一套"工具箱"：
- 里面有40多种工具
- AI知道每个工具是干嘛的
- 你说"创建创意"，AI就从工具箱里拿出"创建工具"

**MCP协议** 就像给AI装了一个"万能插座"：
- 可以插各种外部服务
- 比如天气API、高铁查询、邮件发送等
- 插上就能用，不需要改AI的代码

**查高铁票**：
- ❌ 现在不能直接查（工具箱里没有这个工具）
- ✅ 可以通过MCP接入（插一个高铁查询的插座）
- ⚠️ 需要你自己配置MCP Server

- 传统AI：你问它答，只能聊天
- 这个Agent：你给任务，它**真正干活**

- 简单脚本：按固定流程执行

#### 工具权限分级

| 权限级别 | 工具示例 | 行为 |
|----------|----------|------|
| read（只读） | search_creativity, get_weather | 自动执行 |
| write（写入） | create_creativity, update_settings | 需要确认 |
| dangerous（危险） | permanent_delete, batch_delete | **禁止执行** |

#### 大白话解释

AI有一套**"安全刹车系统"**：

**第1道刹车：工具分类**
- 安全工具（查天气）→ 直接执行
- 危险工具（删除数据）→ 必须你确认
- 超危险工具（永久删除）→ **直接禁止**

**第2道刹车：关键词检测**
- 你说"删除所有" → 触发警报
- 你说"格式化磁盘" → 触发警报
- AI会拒绝执行危险指令

**第3道刹车：沙箱隔离**
- AI执行代码时，不能访问你的文件
- 不能加载外部模块
- 只能用安全的数学和字符串函数

**第4道刹车：目录限制**
- AI只能访问你允许的文件夹
- 不能随意读取敏感文件

**总结**：这个AI助手**不会伤害你**，有多重安全机制保护。

## 三、系统缺陷清单

### 3.1 严重缺陷（P0 - 必须修复）

| 缺陷 | 影响 | 大白话解释 |
|-----|------|-----------|
| **聊天记录存在localStorage** | 数据易丢失 | 浏览器存储只有2MB，存满了就存不进去，而且重启可能丢 |
| **记忆系统无分层设计** | 无法区分短期/长期记忆 | 所有记忆混在一起，没有"刚才说的"和"一直记住的"区分 |
| **Anthropic Tool Calling适配不完善** | Claude用户功能受限 | Claude模型不能用工具，只能纯聊天 |
| **RAG不会自动更新索引** | 创意变更后检索不准 | 你修改了创意，但AI搜的还是旧内容 |
| **聊天记录无100条限制** | 内存溢出风险 | 对话多了会卡，甚至崩溃 |

### 3.2 中等缺陷（P1 - 建议修复）

| 缺陷 | 影响 | 大白话解释 |
|-----|------|-----------|
| **自定义人格无UI** | 用户无法切换AI角色 | 代码里有人格系统，但界面上用不了 |
| **无Prompt模板库** | 每次从零写提示词 | 没有预设的"帮我写小说"、"总结这段文字"等快捷指令 |
| **无对话管理功能** | 历史对话难查找 | 不能搜索、分类、导出以前的聊天记录 |
| **Agent工作流只有5个预设** | 复杂任务受限 | 不能自定义AI的工作流程 |
| **无AI反馈机制** | 无法改进回复质量 | 用户不能点赞/点踩来告诉AI好坏 |
| **技能系统无自动触发** | 技能不会自动激活 | AI不会根据对话内容自动调用相关技能 |

### 3.3 轻微缺陷（P2 - 可优化）

| 缺陷 | 说明 |
|-----|------|
| 代码中有`@ts-nocheck` | 跳过了TypeScript检查，可能有隐藏bug |
| 错误处理不够完善 | 有些地方出错只是console.log，用户看不到提示 |
| 缺少单元测试 | 核心功能没有自动化测试 |
| 文档不够详细 | 开发者文档缺失，新手上手难 |
